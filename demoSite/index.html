How RLS Works

Service Key (KEYS.masterKey) — bypasses all RLS. Only key that can manage _rls endpoints and table/schema operations.

JWT Tokens — HMAC-SHA256 signed. Secret lives in PropertiesService (server-side only). Caller can't spoof user identity. Each token:

  • Contains sub (user ID), iat, exp (24h expiry)
  • Sent as Authorization: Bearer <token> header
  • Verified on every request — signature, expiry checked

API Users — stored in __users sheet (_id | gperms | tables). Each has:

  • gperms — global permission: r (read-only) or w (read+write)
  • tables — per-table overrides, e.g. {"members":"w"} overrides gperm for that table

Row-Level Security (Supabase-style):

  • Every table has an owner_id column — stamped server-side on INSERT from the verified JWT
  • Caller CANNOT set or spoof owner_id — it's always derived from the token
  • SELECT / UPDATE / DELETE: owner_id = auth.uid() is unconditionally ANDed onto the user's WHERE clause
  • No query can escape the policy — enforced inside CRUD, not as a post-filter
  • Service key bypasses all policies (sees all rows, can write any row)

RLS Toggle — when RLS is off, all valid tokens get full read+write access (backward-compatible escape hatch).
